# 設計理念（Design Principles）

## 一、核心主題（Theme）
「時間中的音樂（Music in Time）」

本專案以「時間」作為主要閱讀軸線，將作曲家生平、代表作品與歷史背景整合在同一個探索介面中。目標不是單純展示資料，而是讓使用者在脈絡中理解音樂如何形成、轉變與被時代影響。

---

## 二、核心目標（Objectives）

### 1. 用時間建立理解框架
音樂史常被切成作曲家介紹、作品列表與背景補充三個獨立區塊，初學者很難把它們串起來。

本系統希望把以下資訊放進同一條時間軸：
- 生平事件
- 作品創作或發表年份
- 歷史與文化背景
- 音樂風格轉變

讓使用者能直接看見「什麼時候發生了什麼事」，以及「這些事情如何影響作品」。

### 2. 呈現創作演化而非零散作品
本專案強調觀察作曲家創作歷程，而不是只停留在名曲清單。

因此資料設計與視覺呈現都應支援：
- 早期／中期／晚期分期
- 作品類型分布
- 創作高峰與轉折點
- 年齡與作品之間的對照

### 3. 把作品放回歷史脈絡中理解
作品不應被孤立閱讀。除作品本身外，也要補上：
- 歷史事件
- 社會與文化環境
- 音樂技法與風格發展
- 同時代審美與制度條件

系統的核心問題是：
「這首作品為什麼會在這個時間點出現？」

### 4. 降低音樂史的理解門檻
本專案以多模態方式降低抽象門檻：
- 視覺：時間軸、內容卡片、類型標記、背景區塊
- 文字：簡介、背景、摘要與導讀
- 媒體：影片連結或嵌入播放

### 5. 支援探索式學習
本系統不預設唯一閱讀順序，而是鼓勵使用者自由探索。

主要互動包括：
- 點擊作品卡片切換內容預覽
- 查看事件與背景對照
- 以作品類型與時期進行比較
- 透過影片或外部連結延伸理解

---

## 三、目標使用者（Target Users）

### 1. 初學者
- 需要結構化音樂史導讀
- 需要繁體中文介面與清楚的背景說明

### 2. 音樂愛好者
- 關注作品演化、風格差異與歷史脈絡
- 希望從代表作品延伸到較完整的創作地圖

### 3. 教學場景
- 音樂史課堂導讀
- 報告、展示與課堂輔助工具

---

## 四、核心設計原則（Design Principles）

### 1. Time-first Design
所有內容以時間作為主要索引，年份不是附屬資訊，而是整個介面的主幹。

### 2. Linked Understanding
時間軸、內容預覽與背景區塊必須互相對應，不能只是三塊並列資訊。

### 3. Context as Meaning
背景資訊不是裝飾，而是理解作品的必要條件。

### 4. Bilingual Clarity
介面以繁體中文為主，作品名、作曲家名與重要術語採「繁中 + 原文」，同時兼顧可讀性與專業性。

### 5. Progressive Disclosure
先給使用者必要資訊，再逐步展開更深的內容，避免一開始資訊過載。

### 6. Data for Interpretation
資料欄位不只為了儲存資訊，也要能支援解讀，例如：
- `period`
- `type`
- `age`
- `summary`
- `media`

這些欄位都應服務理解與展示，而不只是存在於 JSON 中。

## 五、目前專案已落地的方向（Current Direction）

目前專案已經形成以下具體方向：
- 採靜態網站架構，部署目標為 GitHub Pages
- 前端以 HTML、Vanilla JavaScript 與 Tailwind utility class 為主
- 使用 JSON 儲存作曲家、作品、記事與背景資料
- 已先以 Beethoven 作為第一位完整示範作曲家
- 已開始將作品、事件與年齡資訊納入資料模型
- 媒體設計採「可嵌入則嵌入，否則導向 YouTube 原站」的降級策略

---

## 六、差異化定位（Differentiation）

本系統不是一般作品清單，也不是單純播放器，而是整合以下三個維度：
1. 時間脈絡
2. 作品分布與創作演化
3. 歷史與文化背景

差異化價值在於：
- 讓音樂史變得可探索
- 讓作品與人生階段建立連結
- 讓學習過程從背資料轉向理解脈絡

---

## 七、設計關鍵問題（Design Questions）

本產品持續要回答：
1. 使用者如何快速理解作品與年份的關係？
2. 如何讓「年齡」、「時期」與「作品類型」形成有效對照？
3. 如何讓背景資訊成為理解作品的支撐，而不是旁枝資訊？
4. 如何在原型階段先建立穩定結構，再逐步升級為更完整的資料視覺化體驗？

---

## 八、警告與注意事項（Warnings）

### 1. 不要讓資料越來越多，但介面沒有更清楚
資料量增加後，若沒有排序、分層與篩選機制，時間軸很快會失去可讀性。

### 2. 不要把背景資訊做成純補充欄
背景資料必須能對應作品與生平，否則使用者不會真的理解它的重要性。

### 3. 不要忽略資料欄位的一致性
若 `type`、`period`、`age`、`media` 等欄位命名不一致，前端顯示與後續擴充都會變得脆弱。

### 4. 不要假設所有 YouTube 影片都能嵌入
影片嵌入有版權與平台限制，前端必須保留外部連結的降級機制。

### 5. 不要太早把原型做成大型框架系統
目前專案重點是先把資料、結構與閱讀體驗做好，再決定是否升級建置流程或導入更完整框架。
