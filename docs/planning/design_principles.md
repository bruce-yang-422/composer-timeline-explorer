# 設計理念（Design Principles）

## 一、核心主題
「時間中的音樂（Timeline of Music）」

本專案以時間作為理解音樂的主軸，但不是把時間軸當成單純的年表，而是把「作曲家個人資料、生平記事、作品、時代背景」放進同一條閱讀路徑。  
目前系統已從單純作品導向，進一步發展成「先看人，再看事件與作品」的閱讀型原型。

---

## 二、目前產品方向

### 1. 以時間建立理解框架
核心資訊需要能在同一脈絡下閱讀：
- 作曲家個人資料
- 生平事件
- 作品創作或發表年份
- 作曲家當時年齡
- 歷史與文化背景

### 2. 以人物作為進入點
目前首頁預設不是先落在某首作品，而是先顯示作曲家個人資料。  
這代表本系統的入口邏輯是：
1. 先理解這是誰
2. 再理解他經歷了什麼
3. 最後把作品放回人生與時代中閱讀

### 3. 從名曲清單走向創作歷程
作品不只是「可點的曲名列表」，而是要和下列資訊互相對照：
- 早期／中期／晚期分期
- 作品類型
- 生平轉折
- 年齡
- 時代背景

### 4. 背景資訊必須服務理解
背景資訊不是裝飾，也不是附錄，而是要回答：
「這首作品為什麼會在這個時間點出現？」

### 5. 以繁中介面降低理解門檻
目前方向維持清楚：
- UI 以繁體中文為主
- 作曲家名、作品名、重要術語採繁中 + 原文
- 文案優先服務閱讀與導覽，而不是百科式堆疊

---

## 三、目前設計原則

### 1. Time-first
年份不是附屬資訊，而是整體架構的骨幹。  
時間軸、甘特分布、背景篩選都必須能回到年份與生命歷程。

### 2. Person-first Reading
系統入口優先是作曲家本人，而不是單一作品。  
個人資料卡在時間軸中應與一般生平記事明顯區隔，作為閱讀入口卡存在。

### 3. Context-first Interpretation
作品不能脫離生平與時代背景單獨理解。  
個人資料模式、事件模式、作品模式，都應有對應的背景層顯示邏輯。

### 4. Progressive Disclosure
先呈現必要資訊，再逐步展開導聆、分析、學術解析與延伸資料，避免一次塞滿所有內容。

### 5. Data for Reading, not Just Storage
目前資料欄位設計已不只是儲存用途，而是直接服務介面展示與閱讀：
- `summary`
- `timelineTitle`
- `portrait`
- `media`
- `sources`
- `age`
- `period`

### 6. Distinct Modes, Distinct UI
個人資料、事件、作品三種模式不應共用完全相同的顯示語言。  
若使用者點的是「個人資料」，就不應殘留作品導聆；若點的是事件，就不應顯示作品型說明。

### 7. Source Traceability
資料可以先不在前台全面顯示來源，但資料層應保留可追溯性。  
因此 `works/*.json` 與 `events/*.json` 已加入 `sources`，而 `contexts/all.json` 則作為後續背景註記、校對與擴充基礎。

### 8. Graceful Media Fallback
外部媒體不可控，因此系統設計必須允許：
- 可嵌入則播放
- 不可嵌入則外部開啟
- 無媒體則正常閱讀，不讓內容失效

## 四、目前已對齊的實作方向

目前 repo 已經落地的方向包括：
- 靜態網站架構
- GitHub Pages 作為部署目標
- Beethoven 作為第一位完整示範作曲家
- 作曲家個人資料模式
- 時間軸中的個人資料入口卡
- 生平事件與作品雙欄時間軸
- 作品類型與時期篩選
- 互動式甘特分布圖
- 來源欄位與肖像欄位的資料設計

- 第二位作曲家資料導入
- 更完整的 type mapping 結構
- 來源資訊在前台的呈現策略

---

## 六、目前文件應維持的共識

1. 文件必須反映目前 repo 的真實狀態，不描述尚未完成的功能為既成事實。
2. 資料欄位設計要優先考慮閱讀與擴充，而不只是儲存。
3. 專案核心價值是「脈絡理解」，不是作品播放器或單純索引。
4. 增加資料時要同步考慮介面負擔，避免資訊量成長快於畫面整理速度。

---

## 七、注意事項

### 1. 不要讓資料量超過介面分層能力
如果只持續增加作品與事件，而沒有清楚的模式切換與視覺層級，時間軸會很快失去可讀性。

### 2. 不要把目前甘特圖誤寫成最終視覺化系統
目前已有可操作的分布圖，但仍屬第一版，不應在文件中描述成完整資料視覺化成果。

### 3. 不要忽略來源欄位的長期價值
即使前台暫不顯示 `sources`，這些欄位已是後續校對、擴充與內容治理的重要基礎。

### 4. 不要讓模式切換彼此殘留
個人資料、事件、作品三種模式需要清楚切換，避免畫面上殘留錯誤內容。
