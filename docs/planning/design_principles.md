# 設計理念（Design Principles）

## 一、核心主題
「時間中的音樂（Music in Time）」

本專案以「時間」作為主要閱讀軸線，將作曲家生平、代表作品與歷史背景整合在同一個探索介面中。現階段目標不是一次完成完整資料視覺化，而是先建立一個可操作、可閱讀、可逐步擴充的原型系統。

---

## 二、目前已確認的產品方向

### 1. 以時間建立理解框架
本系統以年份作為主要索引，讓使用者能把以下資訊放在同一個理解脈絡中：
- 生平事件
- 作品創作或發表年份
- 時代背景
- 音樂風格轉變
- 作曲家當時年齡

### 2. 從「名曲清單」走向「創作歷程」
目前資料設計已開始支援：
- 早期／中期／晚期分期
- 作品類型區分
- 年齡與作品關聯
- 創作轉折與人生事件對照

### 3. 背景資訊必須服務理解
背景資訊不是裝飾，也不是附錄，而是用來回答：
「這首作品為什麼會在這個時間點出現？」

### 4. 以繁中介面降低理解門檻
目前方向已明確：
- UI 以繁體中文為主
- 作曲家名、作品名、重要術語採繁中 + 原文
- 文字內容以導讀與理解為優先，而不是百科堆疊

### 5. 先做可用原型，再升級視覺化
目前策略不是先追求大型圖表，而是先建立：
- 穩定資料模型
- 可操作的三欄介面
- 可點擊作品預覽
- 可顯示背景資訊
- 可處理影片 fallback

---

## 三、目前設計原則

### 1. Time-first
所有核心內容都以時間為主索引，年份不是附屬資訊，而是整體架構主幹。

### 2. Context-first Interpretation
作品不能脫離生平與時代背景單獨理解。

### 3. Bilingual Clarity
顯示內容優先可讀性，但保留原文對照，兼顧教學與研究使用。

### 4. Progressive Disclosure
先呈現必要資訊，再逐步展開背景、媒體與延伸內容，避免資訊過載。

### 5. Data for Reading, not Just Storage
目前 `period`、`type`、`age`、`summary`、`media` 這些欄位，都是為了支援閱讀與介面展示，而不是只為了存資料。

### 6. Graceful Fallback
外部媒體不可控，因此設計上必須允許：
- 可嵌入則播放
- 不可嵌入則外部開啟
- 無媒體則顯示提示

---

## 四、目前實作已對齊的方向

目前專案已落地的部分包括：
- 靜態網站架構
- GitHub Pages 作為部署目標
- Beethoven 作為第一位完整示範作曲家
- composer / works / events / contexts JSON 資料結構
- 作品與事件年齡欄位
- 作品卡片點擊後更新內容預覽
- YouTube fallback 顯示邏輯
- 以 Tailwind utility class 為主的前端原型樣式方向

## 六、目前文件應持續維持的共識

1. 先確保資料與閱讀邏輯穩定，再追求更複雜視覺化。
2. 文件要反映目前原型狀態，而不是假設已完成所有功能。
3. 資料欄位設計應優先考慮展示用途與後續擴充性。
4. 專案核心價值是「脈絡理解」，不是單純播放器或作品索引。

---

## 七、注意事項

### 1. 不要讓資料成長速度超過介面整理速度
若持續增加事件與作品，但沒有排序、分層與篩選，原型會很快失去可讀性。

### 2. 不要把目前保留區寫成已完成圖表
中央區塊目前仍是保留區，不應在文件中描述成已完成 Gantt 或完整視覺化。

### 3. 不要忽略 fallback 設計的重要性
YouTube 嵌入限制已經證明媒體不可完全控制，這不是例外，而是系統設計的一部分。

### 4. 不要讓規劃文件脫離實作現況
文件必須持續對齊目前 repo 的真實狀態，否則後續會失去參考價值。
